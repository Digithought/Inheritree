Findings marked Verified were reproduced by executing probe scripts against the built dist/ output; the probe script and its raw results are in the appendix.

Summary

This is a genuinely small, fast, well-shaped B+Tree. The node layout (flat 64-wide arrays, instanceof dispatch, splice-based mutation), the versioned-path invalidation scheme, and the split/rebalance logic are all sound — a stress probe of 5,000 inserts followed by 2,500 interleaved deletes kept ordering and count intact. The review found one significant correctness bug (with a second symptom in range()), a handful of API sharp edges, and several worthwhile performance and cleanliness improvements.

1. Correctness bugs

1.1 next() from an end-of-leaf crack never advances Verified

b-tree.ts:347–380 (internalNext)

When find(key) misses, it returns a crack at the insertion point. If that insertion point is the end of a leaf (the sought key falls between the last entry of one leaf and the first entry of the next), internalNext gets a path with on === false and leafIndex === entries.length. The crack-recovery block sets on back to false (index out of range) and then returns without advancing, because the leaf-advance logic lives in the else chain that is only reachable when on === true:

private internalNext(path: Path<TKey, TEntry>) {
    if (!path.on) {  // Attempt to move off of crack
        path.on = path.branches.every(...) && path.leafIndex < path.leafNode.entries.length;
        if (path.on) return;
        // <-- falls out of the function here; the "advance to next leaf" logic below is unreachable
    } else if (path.leafIndex >= path.leafNode.entries.length - 1) {
        ... // pops branches and moves to the first entry of the next leaf
    }
    ...
}

Reproduction (NodeCapacity = 64, so 65 inserts force one split; leaves become 0–310 and 320–640):

const tree = new BTree<number, number>();
for (let i = 0; i <= 64; i++) tree.insert(i * 10);
const crack = tree.find(315);          // crack at end of first leaf
tree.at(tree.next(crack));             // → undefined   (expected 320)
tree.at(tree.next(tree.find(15)));     // → 20          (mid-leaf crack works)

The README explicitly advertises "Find nearest, using next on an unsuccessful path", and internalPrior handles its mirror case correctly (prior(find(315)) → 310), so this is an asymmetry bug, not a design decision. It silently produces wrong results — no exception — which is the worst failure mode for a storage primitive.

Suggested fix: when crack recovery fails because the crack sits at/after the end of the leaf, fall through to the same pop-and-advance logic used for the last-entry case instead of returning. Sketch:

if (!path.on) {
    path.on = path.branches.every(b => b.index >= 0 && b.index < b.node.nodes.length)
        && path.leafIndex >= 0 && path.leafIndex < path.leafNode.entries.length;
    if (path.on || path.leafIndex < path.leafNode.entries.length) {
        return;   // recovered onto an entry, or crack precedes an entry in this leaf
    }
    // end-of-leaf crack: fall through to advance into the next leaf
}
if (path.leafIndex >= path.leafNode.entries.length - (path.on ? 1 : 0)) {
    ... existing pop/moveToFirst logic (moveToFirst already sets leafIndex/on correctly)
} else {
    ++path.leafIndex;
    path.on = true;
}

Add regression tests for cracks in all four positions: before the first entry of the tree, mid-leaf, between two leaves (this bug), and after the last entry of the tree — for both next and prior.

1.2 range() returns empty for a start bound on an end-of-leaf crack Verified

b-tree.ts:266–276 (findFirst), b-tree.ts:62–84 (range)

Same root cause, second public symptom. findFirst uses internalNext to step off a missed start bound, so a range whose (non-matching) start key lands between leaves begins iteration on an off path, and the loop's !path.on guard exits immediately:

// leaves are 0–310 and 320–640
[...tree.range(new KeyRange(new KeyBound(315), new KeyBound(345)))]
// → []   (expected paths to 320, 330, 340)

Fixing 1.1 fixes this too, but range() deserves its own regression tests with bounds straddling leaf boundaries in both directions (a descending range with a missed last bound exercises the same code path via findLast).

1.3 Redundant partition update in leaf merge (harmless, but worth a comment)

b-tree.ts:618–620 (rebalanceLeaf, merge-right case)

When merging the right sibling into the leaf, the pIndex === 0 branch re-writes an ancestor partition to keyFromEntry(leaf.entries[0]). Since a non-root leaf can never be empty at this point (underflow threshold is 32) and a deletion at index 0 already updated ancestors at b-tree.ts:434, and merging right never changes entries[0], this write appears to always store the value already present. Not a bug — but partition maintenance is the subtlest part of this file, and the existing comment at line 676 shows a corruption bug already lived here once. Either delete the redundant call with a comment explaining why it's unnecessary, or keep it and note it's defensive. Better yet, see the invariant-checker suggestion in §6.

2. Robustness & API sharp edges

2.1 Iterators yield one shared, mutated Path object Verified

b-tree.ts:250–264 (internalAscending/internalDescending)

The generators yield the same path instance every iteration and mutate it in place. Anyone who does the natural thing —

const paths = [...tree.ascending(tree.first())];
paths.map(p => tree.at(p));   // → [undefined, undefined, undefined]

— gets N references to a single cursor parked off the end of the tree. The README examples always consume tree.at(path) inside the loop, so this works until a user deviates slightly. Options, in order of preference:

  • Provide entry-yielding convenience iterators (entries(range?), keys(), [Symbol.iterator]) as the documented default — they sidestep the aliasing entirely and remove the tree.at(path) ceremony from the 95% case. Keep the path-yielding iterators for cursor-level work.
  • Or yield path.clone() per step (costs one small allocation per element).
  • At minimum, document loudly that yielded paths are live and must be cloned to retain.

2.2 The comparator consistency check doubles comparator calls and catches little

b-tree.ts:242–248 (compareKeys)

const result = this.compare(a, b);
if (result !== 0 && result === this.compare(b, a)) {
    throw new Error("Inconsistent comparison function for given values");
}

Two problems:

  • Cost: every non-equal comparison invokes the user comparator twice. Measured: a single find in a 1,000-entry tree makes 19 comparator invocations for ~10 logical comparisons. For expensive comparators (string collation, composite keys) this roughly halves search throughput of the hottest path in the library.
  • Coverage: the check only fires when both directions return the identical non-zero value. It misses compare(a,b) === -1 with compare(b,a) === -2 (fine in practice) but also misses compare(a,b) === -1 with compare(b,a) === 0 — a genuine inconsistency. So the tree pays double for a partial guarantee.

Suggestion: make the check opt-in (constructor option or a DebugBTree subclass — the method is already protected and documented as overridable, so the machinery exists; flip the default). A cheaper always-on alternative: validate the comparator once with a few sentinel probes at construction, or check only the first N comparisons.

2.3 Failed insert freezes the caller's object Verified

b-tree.ts:95–102

insert calls Object.freeze(entry) before checking for a key conflict. On a duplicate, the insert is rejected but the caller's object is now permanently frozen — a surprising side effect on a failed operation (verified: Object.isFrozen(dup) === true after a rejected insert). Move the freeze after the conflict check (i.e., freeze inside internalInsertAt, or only when path.on comes back true).

2.4 updateAt on an off-entry path returns [path, true]

b-tree.ts:104–122, 412–428

If the given path is not on an entry, nothing happens, yet the result says wasUpdate: true. The doc comment does explain this encoding, but "it was an update" for an operation that did nothing is a trap; callers must check path.on and interpret wasUpdate jointly. Consider a three-state result, throwing on an off-entry path (it's a caller error, and the library already throws for stale paths), or at least renaming the doc so the failure mode is impossible to misread.

2.5 deleteAt leaves the caller with no usable path

b-tree.ts:163–170

The rebalancing code carefully keeps the passed path coherent (adjusting leafNode/leafIndex through borrows and merges), but deleteAt bumps _version without stamping it onto the path — so the one path that is still positionally correct is unusable. Setting path.version = ++this._version would hand back a valid crack at the deletion point, enabling the very common "delete while iterating" pattern (deleteAt then moveNext) without a re-find. This mirrors what insert/updateAt/upsert already do for their returned paths.

3. Performance opportunities

Roughly in order of expected payoff:

  1. Comparator double-call (§2.2) — the single biggest lever; it sits inside every binary-search probe at every tree level.
  2. Optional freezing. Object.freeze on every insert costs real time in bulk loads (hidden-class transition + the call itself) while providing only shallow protection the README already disclaims. A constructor option ({ freeze: false }) keeps safety-by-default and gives bulk loaders an out.
  3. Iterative descent. getPath, getFirst, getLast recurse to the leaf and then unshift each PathBranch on the way back up — O(depth²) element moves plus a frame per level. A simple loop pushing into a preallocated array is flatter and allocation-friendlier. Similarly, path.branches.splice(-popCount, popCount) in internalNext/internalPrior allocates the removed-elements array on every leaf transition; path.branches.length -= popCount is free.
  4. range() end test. The loop calls the user comparator (twice, per §2.2) for every yielded element to test the end bound. Since the end is a fixed position and iteration is strictly sequential, comparing positions is sufficient and free: capture endPath.leafNode/endPath.leafIndex, yield until the current path matches, then stop. This turns range scans from O(n·cmp) to O(n).
  5. Bulk load. Building from a sorted array bottom-up (fill leaves to ~full, then build branch levels) is O(n) versus O(n log n) repeated inserts, and produces better fill factor. A natural fit for the stated "helper add-on" philosophy if you'd rather keep the core minimal — but it needs internal access, so it belongs here.
  6. Optional O(1) count. A single _count field maintained in internalInsertAt/internalDelete is one add per mutation and would make getCount() free for the no-argument case. The partial-count overload can keep the current walking behavior. Minor memory, large ergonomic win; many users reach for size first.

Things that look tempting but are probably fine as-is: splice-based array mutation on 64-wide nodes is cache-friendly and fast in V8; instanceof dispatch is already benchmarked (per the comment in nodes.ts); NodeCapacity=64 fixed is a reasonable call.

4. Code cleanliness

4.1 Type safety: ITreeNode is an empty interface

nodes.ts:2, casts throughout b-tree.ts

An empty interface matches everything in TypeScript (const n: ITreeNode = 42 compiles), so the many node as LeafNode<TEntry> / as BranchNode<TKey> casts are unchecked. A discriminated union removes every cast for free, because instanceof narrows unions natively:

export type TreeNode<TKey, TEntry> = LeafNode<TEntry> | BranchNode<TKey, TEntry>;

export class BranchNode<TKey, TEntry> {
    constructor(
        public partitions: TKey[],
        public nodes: TreeNode<TKey, TEntry>[],   // children now typed
    ) { }
}

Then if (node instanceof LeafNode) { node.entries ... } else { node.partitions ... } type-checks with zero casts and zero runtime change. This is the highest-value cleanliness fix in the codebase.

4.2 Duplicated mirror-image code

  • getFirst/getLast (b-tree.ts:505–530) duplicate moveToFirst/moveToLast (476–502). first() can be: construct a blank path, call moveToFirst(this._root, path). Two functions deleted, one code path to maintain.
  • indexOfEntry/indexOfKey (305–345) are the same binary search differing only in key extraction and the equal-case return. Acceptable duplication for hot-path reasons, but worth a shared comment cross-referencing them so a fix to one isn't forgotten in the other.
  • internalNext/internalPrior, findFirst/findLast, rebalanceLeaf/rebalanceBranch are inherent mirror pairs — fine to keep separate for clarity/speed, but note that internalPrior validates the path (line 383) while internalNext does not, and callers like movePrior then validate twice. Pick one convention (validation belongs in the public wrappers, which already do it).

4.3 Smaller items

  • NodeCapacity >>> 1 appears seven times as the underflow threshold; a named const HalfCapacity states intent.
  • findFirst guards range.first && !range.first.inclusive immediately after asserting range.first! — the extra check is dead; same in findLast.
  • Errors are generic Errors. Exported error classes (or at least error codes) let callers distinguish "stale path" from "inconsistent comparator" programmatically.
  • index.ts doesn't export nodes.ts, yet the public Path.leafNode is typed as LeafNode — the public surface references a type users can't import from the package root. Either export it or (better, per §5.1) stop exposing it.
  • Missing semicolons at b-tree.ts:267 and 279 (const startPath = this.find(...)) — trivia, but the rest of the file is consistent.

5. Design observations

5.1 The Path abstraction leaks — and the README already knows it

All five Path fields are public and mutable, and correctness depends on users not touching them ("Do not change the properties of this object directly"). The "Help wanted" section lists "Better insulation of path's internals," so this is just a vote plus a concrete shape: export a Path interface exposing only on (readonly) and isEqual/clone, keep the implementation class module-private, and have BTree cast internally. Zero runtime cost, full insulation at the type level, no breaking change for correct code. (Determined users can still cast — this is TypeScript — but the accident becomes impossible.)

5.2 Ergonomics of the cursor-only API

Every read goes path-first: tree.at(tree.next(tree.find(k))), for (const p of tree.ascending(tree.first())) use(tree.at(p)). The cursor model is the right foundation — it's what makes find-nearest and range scans composable — but the everyday operations deserve direct forms:

  • ascending()/descending() with no argument (defaulting to first()/last());
  • entries(range?) yielding TEntry directly and [Symbol.iterator] so for (const e of tree) works — this also neutralizes the aliasing trap of §2.1;
  • clear() — currently the only way to empty a tree is to delete every entry or discard it.

These are thin wrappers, not scope creep, and they'd let the README examples shrink.

5.3 Versioning model is coarse but honest

A single global _version invalidates every outstanding path on any mutation, even when structurally unnecessary (an in-place upsert of an existing key moves nothing). That's a defensible trade — precise invalidation would cost bookkeeping on the hot path — but two consequences are worth documenting: iterators cannot survive any mutation, including updates that don't move entries; and the delete-while-iterating pattern requires a re-find per deletion, making it O(n log n) (§2.5 would fix this). Note also that _version is an unbounded counter incremented per mutation; at ~9×1015 mutations doubles lose integer precision. Not a practical concern in-memory, but a one-line comment (or wrapping) costs nothing.

5.4 What the design gets right

Worth stating so the criticism above has proportion: storing data only in leaves with an open path structure instead of leaf links is well-reasoned for a single-threaded in-memory tree (the README's justification is correct); the split algorithm's indexDelta propagation keeping the insertion path valid through cascading splits is elegant; freezing entries by default is the right default for a structure whose integrity depends on key immutability; and the decision to keep NodeCapacity fixed rather than configurable is a good simplicity/performance call. The test suite is stronger than typical for a project this size (branch-rebalance, delete-integrity, path-invalidation, and invariants suites all exist).

6. Testing recommendations

  • Randomized oracle test. The single most valuable addition: run 10⁵–10⁶ random operations (insert/delete/updateAt/upsert/merge/find/range with random keys) mirrored against a sorted array, comparing results after every operation, with several seeds. This class of test finds bug 1.1 in seconds — find of a random absent key followed by next disagrees with the oracle as soon as a leaf boundary is hit — and permanently guards the rebalancing logic, which is exactly where the previously-fixed partition-corruption bug (comment at b-tree.ts:676) lived.
  • Structural invariant checker. A test-only checkInvariants(tree) walking the tree asserting: all leaves at equal depth; every non-root node ≥ half full; partitions.length === nodes.length - 1 in every branch; every partition equals the minimum key of its right subtree; entries strictly ordered by the comparator. Call it inside the oracle test after every mutation. This makes redundancies like §1.3 provable and future rebalance edits safe. (invariants.test.ts exists — extending it with the partition-equals-subtree-min check is the key addition.)
  • Crack-position matrix. Explicit tests for next/prior/range from cracks at: tree start, mid-leaf, leaf boundary (bug 1.1), and tree end — at 1-leaf, 2-level, and 3-level tree sizes.
  • The benchmark suite from "Help wanted". Before optimizing anything in §3, a small benchmark harness (bulk ascending insert, random insert, random find, range scan, mixed churn — with number and string keys) makes the comparator-double-call and freeze costs measurable rather than argued. tinybench is a zero-dep fit for the project's philosophy.
