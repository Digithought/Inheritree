Inheritree Code Review

Focus: copy-on-write inheritance layer (the delta over the Digitree base) — correctness, performance, robustness, and design. Originally reviewed 2026-07-06 at 0fcf2ba; revised 2026-07-07 after merging upstream Digitree v1.5.0 (merge commit f4b41a7, package v0.5.0). Baseline: all 285 tests passing.

Verdict

The COW machinery is structurally sound. The ownership model (per-node tree back-pointer, clone-rootward on first write, upward-closed ownership) is coherent, the path-remapping through mutableLeaf/mutableBranch/replaceRootward is careful, and the test suite — particularly assertOwnershipInvariant with base snapshots — encodes the right invariants and exercises the hard rebalance paths. No structural COW-correctness bug was found in this review, and the invariants survived the v1.5.0 merge intact.

The upstream v1.5.0 merge resolved four of the original fourteen findings outright (freeze-on-success, deleteAt path re-stamp, aliasing-free iteration, O(1) count) and substantially improved a fifth (comparator-check cost). The most significant remaining issue is unchanged and now stands alone at the top: node cloning uses structuredClone, which deep-copies user entries — throwing on entries containing functions, silently stripping class prototypes, breaking entry identity, and producing unfrozen copies. All of that is unique to derived trees and all avoidable. Second tier: F3 and F4 (the COW plumbing's allocation overhead on plain trees, and mutableBranch's wasted work on an already-owned spine) are now resolved by the cow-lazy-mutable-node change — the mutable-node layer is allocation-free unless a clone genuinely occurs (delete-heavy benchmarks +2.5–8.6%, no regression). The base-immutability contract remains documented rather than enforced.

Resolved by the v1.5.0 Merge

Resolved F6, F7, F9, F12 — closed upstream; F12 extended for COW in the merge

  • F6 — freeze on success only. insert now calls freezeEntry only after the entry is actually in the tree (src/b-tree.ts, insert); a rejected duplicate no longer freezes the caller's object. Upstream went further than the finding asked: freezing is also now optional per tree ({ freeze: false }).
  • F7 — deleteAt re-stamps the path. deleteAt now stamps the bumped version onto the given path, whose position internalDelete keeps coherent — the delete-while-iterating pattern the original finding described is now the documented, tested behavior. This works identically through the COW layer, since mutableLeaf remaps the same path object it re-stamps.
  • F9 — aliasing-free iteration. entries(), keys(), and [Symbol.iterator] yield distinct values (so [...tree] works), and ascending/descending now carry explicit warnings about the reused-cursor behavior. Exactly the remedy option the finding preferred.
  • F12 — O(1) count. Upstream added the stored _count (the "incremental" design from the finding). The merge extends it to inheritance: a derived tree seeds _count from its base's stored count at construction (O(1)) and tracks its own ±1 delta — so size/getCount() are O(1) on children too, which matters for the versioning workloads Inheritree targets. Note the new coupling this creates with the immutability contract (see F2).

Open Findings

High F1 — structuredClone deep-copies user data during COW cloning (unchanged by merge)

src/nodes.ts (LeafNode.clone, BranchNode.clone)

clone(newTree: BTree<any, any>): LeafNode<TEntry> {
    return new LeafNode(structuredClone(this.entries), newTree);
}

When a derived tree first writes to a shared leaf, every entry in that leaf is deep-cloned. Four distinct problems, all empirically verified in Node:

  1. Throws on legal entries. Any entry containing a function, symbol, or other non-cloneable value raises DataCloneError — but only when a derived tree happens to COW-clone the leaf holding it. The same entry works fine in a plain tree, so this surfaces as a data-dependent, late failure.
  2. Silently strips prototypes. A class-instance entry becomes a plain object in the derived tree: methods and getters vanish. No error, just wrong data.
  3. Breaks entry identity. derived.get(k) !== base.get(k) after a COW clone of that leaf, but === for entries in untouched leaves. Identity-based user logic behaves inconsistently depending on internal node boundaries the user can't see.
  4. Un-freezes entries. structuredClone of a frozen object returns an unfrozen copy, undoing the protection freezeEntry applied on insert.

It is also internally inconsistent — the rebalance merge paths move base-owned entries into child-owned leaves by reference, so cross-tree sharing of entries is already the design's own assumption — and it's the slow option: a deep clone of up to 64 entries where a shallow copy suffices.

Fix: shallow-copy — this.entries.slice() / this.partitions.slice(). Entries are treated as immutable by the tree's own contract (frozen by default; with the new freeze: false option the README still forbids post-insert mutation), so sharing them by reference across base and child is exactly as safe as the sharing the tree already does everywhere else. The merge sharpens this finding: with upstream's new { freeze: false } option, entries are never frozen at all, yet structuredClone still pays full deep-copy cost on them — the one configuration that asked for maximum speed gets the slowest clone. Add tests alongside the fix: class-instance entries, function-bearing entries, entry identity across base/derived, frozen-ness of entries in cloned leaves.

Medium F2 — Base-immutability contract has no runtime guard (unchanged by merge; slightly more valuable now)

readme.md "Help wanted"; src/b-tree.ts constructor docs, root getter

Already acknowledged in the readme. The pieces remain in place after the merge: every tree maintains _version, and every public operation funnels through validatePath or the root getter. That makes the guard nearly free:

constructor(..., base?) {
    ...
    this.baseVersion = base?.chainVersion() ?? 0;
}

/** Version of this tree plus everything it inherits from (O(chain), chains are short). */
private chainVersion(): number {
    return this._version + (this.base ? this.base.chainVersion() : 0);
}

private checkBase() {
    if (this.base && this.base.chainVersion() !== this.baseVersion) {
        throw new MutatedBaseError();  // base (or an ancestor base) mutated after derivation
    }
}

Call checkBase() in the root getter and in isValid. That covers every entry point: fresh operations resolve through root, and navigation on existing paths goes through validatePath → isValid. The recursive sum handles multi-level chains. Cost is two integer ops per public call for the common single-level case.

The merge raises the stakes slightly: a child's _count is now seeded from the base at derivation, so a contract violation no longer just risks structural corruption — it also silently skews the child's O(1) count even when the shared nodes happen to survive. One more cheap, observable symptom the guard would convert into a loud error. Two honest limitations to document remain as before: detection happens at the child's next operation, and the post-clearBase() obligation stays unguardable (see F13).

Resolved F3 — COW plumbing taxes plain (non-derived) trees (fixed in cow-lazy-mutable-node)

Resolved: laziness moved into the callee exactly as the fix shape below proposes. mutableLeaf/mutableBranch now take (path, depth[, sib, delta]) and build slice/sibling-segment material only after deciding a clone is required; leafSibPath (and its fabricated leafIndex) is deleted. Plain and already-owned trees allocate nothing.

src/b-tree.ts — leafSibPath/branchSibSegments call sites in rebalanceLeaf/rebalanceBranch; path.branches.slice(...) at every mutableBranch call site

Overhead present on every borrow/merge/split even when this.base === undefined:

  • leafSibPath clones the entire branch array plus a fresh PathImpl, only for mutableLeaf to take the no-base fast path and discard it all.
  • branchSibSegments likewise allocates a cloned segment list per branch borrow/merge.
  • Every mutableBranch call site pre-allocates path.branches.slice(0, depth + 1) before the callee can early-return on !this.base.

Fix shape: make laziness the callee's job. Pass (path, depth) (and for siblings, (path, depth, sib, delta)) and let mutableLeaf/mutableBranch construct slices/sibling segments only after deciding a clone is actually needed. This also removes the fabricated leafIndex in leafSibPath (meaningless in the sibling leaf; unused only by luck of the implementation).

New since the merge: upstream's v1.5.0 mutation internals are leaner than ever (direct node access, no wrappers), so the fork's per-op delta grew — and upstream also shipped a bench/ harness (tinybench). Before fixing, run the bench on this fork vs. upstream Digitree to get the real number for the delete-heavy workload; after fixing, the same run verifies the win. The fix priority rises accordingly: it is now the difference between "Inheritree costs you nothing until you derive" being true or merely aspirational.

Resolved F4 — mutableBranch does wasted work when the spine is already owned (fixed in cow-lazy-mutable-node)

Resolved: the ownership fast path below now guards mutableBranch (if (!this.base || branch.tree === this) return branch;), so an already-owned spine allocates no Map and runs no remap. A code comment at the site records the tripwire — the fast path is correct only while ownership stays upward-closed (assertOwnershipInvariant check 1 is the guard).

src/b-tree.ts — mutableBranch

For a derived tree that has already cloned its write path (the steady state for any write-heavy child), every subsequent mutableBranch call still allocates a Map, calls replaceRootward (which returns immediately at the first owned segment), then runs mainPath.remap — an O(depth) loop of map.get calls against an empty map. mutableLeaf already has the ownership fast path; mutableBranch is missing its twin:

const branch = segments.at(-1)!.node;
if (!this.base || branch.tree === this) {
    return branch;   // owned ⇒ all ancestors owned (upward-closed ownership invariant)
}

The justification is the invariant the test suite itself formalizes (assertOwnershipInvariant check 1). Combined with F3, the entire mutable-node layer becomes allocation-free except when a clone genuinely occurs. Fix both in one refactor.

Low F5 — upsert and merge disagree about on after an insert (upstream kept this behavior in v1.5.0)

src/b-tree.ts — upsert, merge

merge sets path.on = true after inserting; upsert leaves on = false, so tree.at(tree.upsert(x)) returns undefined exactly when the entry was freshly added. Documented, but overloading a positional flag as a semantic result remains a footgun, and the two hybrid-mutation siblings resolving it opposite ways makes it worse. Upstream's v1.5.0 operation-result-contract pass tightened updateAt (it now throws PathNotOnEntryError on a crack path) but left the upsert asymmetry in place — so this is now a deliberate upstream position, and the fork should either adopt it as final or fix it upstream-first to avoid drift. Suggested shape stays: align with merge — always on = true, return [path, wasUpdate].

Low F8 — Comparator antisymmetry check is still incomplete (cost half resolved upstream)

src/b-tree.ts — compareKeys, BTreeOptions.checkComparator

The original finding had two halves. The cost half is resolved: v1.5.0 samples the first 32 comparisons by default and drops the check off the hot path entirely, with { checkComparator: true } restoring the exhaustive mode — a better design than the subclass escape hatch. The completeness half remains: the check still only fires when both directions return the identical nonzero value, missing zero-asymmetry (compare(a,b)=1, compare(b,a)=0) and same-sign results of different magnitude (1 vs 2, common with broken subtraction comparators). A sign-based check catches the whole class at the same cost: throw when Math.sign(this.compare(b, a)) !== -Math.sign(result). Now that the check runs on a bounded sample (or by explicit opt-in), there is no performance argument left against strengthening it.

Info F10 — node.tree back-pointer retains the whole tree-object chain (unchanged by merge)

src/nodes.ts

Ownership is checked purely by identity (node.tree === this), but the pointer retains the full BTree object — comparator and key-extractor closures, and transitively the entire base chain of tree objects, which survives clearBase() for as long as any shared node lives. A dedicated ownership token (readonly owner = Symbol() per tree) gives the same O(1) check while decoupling nodes from tree instances, and removes the BTree<any, any> type erasure from nodes.ts. Small new wrinkle from the merge: clear() now creates an owner-stamped root and drops the base, so a cleared child frees its chain — the token design would make that reasoning unnecessary rather than merely correct.

Info F11 — root getter walks the base chain on every access (unchanged by merge)

src/b-tree.ts — root

Every find/first/last on a derived tree resolves this.base.root recursively — O(chain length) per operation, and chains grow linearly with history in the snapshot-per-version pattern. Under the immutability contract the base's effective root cannot legitimately change after derivation, so caching is sound (this._baseRoot ??= this.base.root). If F2's version guard lands, cache-then-check keeps detection intact.

Info F13 — Pair clearBase() with a true-isolation flatten() (unchanged by merge)

src/b-tree.ts — clearBase; readme "Base immutability contract"

The clearBase docs are excellent and honest about the sharing hazard — but "build a fresh tree and re-insert" outsources an O(n log n) workaround for what the library could do in O(n): walk once, cloning every node not owned by this. The merge hands this finding its implementation for free: upstream's buildFrom is exactly the O(n) bottom-up bulk loader a flatten() needs — BTree.buildFrom(this.entries(), this.keyFromEntry, this.compare) is nearly the whole method (plus carrying over the freeze/checkComparator options). It would also close the F2 guard's post-clearBase blind spot for users who choose it.

New Findings from the Merge

Info N1 — Constructor's third argument is now a BTree | BTreeOptions union

src/b-tree.ts — constructor

Upstream put BTreeOptions in the third position; Inheritree's documented API has the base tree there. The merge resolves the collision by discriminating with instanceof BTree and accepting options as a fourth argument when a base is present. This preserves both parties' call sites and is unambiguous (a base can never be a plain options object), but positional unions age poorly — every future upstream constructor change lands on this same seam. Worth a deliberate decision for the fork's next major: either (a) keep the union as the permanent contract and document it prominently, or (b) migrate to options.base (BTreeOptions<TKey, TEntry> gains an optional base) with the positional form deprecated. Option (b) makes the fork's constructor signature identical to upstream's, shrinking the permanent merge surface to the options interface — cheap insurance given how painful this file's merges are.

Low N2 — COW × new-upstream-feature combinations lack dedicated tests

test/ — coverage gap, not a defect

The merged suite (285 passing) covers upstream's new features on plain trees and the fork's COW behavior on the pre-1.5 feature set, but the cross products are only implicitly covered, and several encode fresh merge decisions that deserve pinning:

  • buildFrom as a base: derive a child from a bulk-loaded tree, mutate, verify ownership invariants and base immutability (bulk-loaded nodes are owner-stamped by the merge — a regression here would silently disable COW cloning).
  • clear() on a derived child: the merge decided clear() drops the base (an empty tree inherits nothing). Test that the base is untouched, the child is empty and re-insertable, and the child no longer shares nodes.
  • Counts on children: size/getCount() after derive (equals base), after child inserts/deletes (base's count unchanged), and through a chain (base → c1 → c2).
  • freeze: false children and the options-passing form: new BTree(k, c, base, { freeze: false }) — the four-argument path introduced by the merge has no direct test.
  • Delete-while-iterating on a child: upstream's new deleteAt re-stamp exercised across a COW clone boundary (the remapped path is the one re-stamped).
  • Oracle test with a COW fork: upstream's randomized oracle (b-tree.oracle.test.ts) runs against a Map oracle on one tree; a variant that forks a child mid-sequence and runs both oracles would be the strongest COW regression net available.

What's Working Well

  • The ownership invariant is explicit and tested. assertOwnershipInvariant (connectivity, no-shared-mutable-node, snapshot-verified base immutability) is exactly the right formalization of the COW design — and it carried through the v1.5.0 merge with only type-level edits, which is what a well-factored invariant should do.
  • The hard-won fixes are documented at the point of subtlety. The comments on branchSibSegments, replaceRootward's owned-ancestor linking, and the rebalance merge partition logic explain why the code is shaped this way and what corruption the naive version caused.
  • Honest user-facing docs. The base-immutability contract and the clearBase caveats state the sharp edges plainly.
  • Upstream convergence is real. Four of this review's original findings (F6, F7, F9, F12) and two readme help-wanted items were independently resolved in Digitree v1.5.0 — the fork is tracking a base that is actively improving in the same directions, which lowers the long-term cost of staying a fork.

Suggested Order of Attack (revised)

  1. F1 — shallow-copy node clones. Small diff, fixes a correctness class, pure speedup, and now also the only remaining High. Add the F1 test list.
  2. N2 — the COW × v1.5.0 test matrix, especially the oracle-with-fork variant. Cheap to write against the freshly-merged code while the merge decisions are still crisp, and it protects everything below.
  3. F2 — version-stamp base guard. Closes the readme's top "help wanted" item; now also protects _count integrity.
  4. F3 + F4 together — restructure mutableLeaf/mutableBranch to take (path, depth[, sib, delta]) and build clone inputs lazily; add the owned-spine fast path. Benchmark before/after with the new bench/ harness.
  5. F13 — flatten() via buildFrom; a few lines now that the bulk loader exists.
  6. F5, F8, F14 remainders, N1 as taste dictates; F5 and N1 are fork-positioning decisions (align-with-upstream vs. fix-and-diverge) more than code tasks.
